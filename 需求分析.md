## 二、幻觉分类体系（最终版）

### 设计原则

按**知识库的认知状态**分类，而非按"编造了什么内容"分类。每一类对应不同的检测策略和业务修复手段。

### 核心分类（4 类）

#### 第一类：事实冲突（Fact Contradiction）

- **定义**：知识库中有明确、具体的事实信息，回复给出了与之矛盾的陈述
- **KB 状态**：有明确事实
- **检测难度**：最低（可直接比对）
- **覆盖原类型**：参数编造、信息编造（部分）、优惠编造、政策编造、政策偏差
- **案例**：h01, h02, h04, h05, h06, h08, h11, h17, h19

#### 第二类：能力越界（Capability Overreach）

- **定义**：知识库明确声明系统不具备某项操作能力，但回复声称已经执行了该操作
- **KB 状态**：明确否定（"未接入""不具备""需人工""不可"）
- **检测难度**：中等（需识别否定声明 + 执行动词）
- **覆盖原类型**：能力越界
- **案例**：h03, h10, h14, h18

#### 第三类：无依据断言（Unsupported Assertion）

- **定义**：知识库对某信息既未确认也未否认（沉默状态），但回复做了肯定性陈述
- **KB 状态**：沉默（"未标注""未提及"）
- **检测难度**：最高（KB 沉默 ≠ 回复错误，但客服不应无依据做肯定陈述）
- **覆盖原类型**：信息编造（KB 沉默的子集）
- **案例**：h09, h15

#### 第四类：安全误导（Safety Misleading）

- **定义**：知识库中有明确的安全警告或禁忌说明，回复给出了相反的、可能造成人身伤害的建议
- **KB 状态**：有安全警告
- **检测难度**：中高（需要领域知识推理链）
- **覆盖原类型**：安全误导
- **案例**：h13

### 附加维度：信息遗漏（Information Omission）

- **定义**：回复本身没有事实错误，但遗漏了知识库中的重要限定/补充信息
- **不计入核心幻觉率**，作为独立维度评估"回复完整性"
- **案例**：h20

### 严重程度分级

| 等级 | 判据 | 案例 |
|------|------|------|
| **高危** | 人身安全、重大金钱损失、法律风险 | h13, h01, h05, h06 |
| **中危** | 关键信息完全错误、能力越界、无依据断言 | h02, h03, h07, h09, h10, h11, h14, h15, h17, h18, h19 |
| **低危** | 部分偏差、信息遗漏 | h04, h08, h20 |

### 边界 case 优先级

当一条回复同时触发多个类型时，按以下优先级归类：

**安全误导 > 能力越界 > 事实冲突 > 无依据断言**

## 三、技术架构

### 整体流程

```
用户上传 replies.json（可选 ground_truth.json）
        │
        ▼
   GUI 界面（Tkinter）
   ├── 文件选择器
   ├── 模式切换（Mock / 真实API）
   ├── API Key 输入框（真实API模式）
   └── 开始检测按钮
        │
        ▼
   检测引擎（逐条处理）
   ├── Mock 模式：规则模拟返回
   └── 真实API模式：调用 LLM API
        │
        ▼
   结果输出
   ├── detection_result.json（整体判断 + 逐句标注）
   └── evaluation_report.json（如有 ground_truth）
```

### 项目文件结构

```
hallucination_detector/
├── main.py                 # 入口：GUI 界面
├── detector.py             # 核心检测逻辑（Prompt + API调用 + 解析）
├── classifier.py           # 幻觉分类体系定义 + 严重程度判定
├── evaluator.py            # 检出率计算（与 ground_truth 比对）
├── mock_detector.py        # Mock 模式模拟逻辑
├── utils.py                # 文件读写、JSON校验、句子拆分等工具函数
├── prompts/
│   └── detection_prompt.py # Prompt 模板
├── output/                 # 检测结果输出目录
└── build.spec              # PyInstaller 打包配置
```

### 轻量化策略

| 策略 | 说明 |
|------|------|
| GUI 用 Tkinter | Python 自带，零额外依赖 |
| API 调用用 `requests` | 替代 `openai` 重型 SDK，直接调 HTTP API |
| 不引入 numpy/pandas | 数据量小（20条），纯 Python 处理足够 |
| `--onefile` 打包 | 单 exe 文件，方便分发 |

## 四、Prompt 工程设计要点

### 输出 JSON Schema（强制约束）

```json
{
  "id": "原始ID",
  "is_hallucination": true/false,
  "hallucination_type": "事实冲突|能力越界|无依据断言|安全误导|null",
  "severity": "高危|中危|低危|null",
  "reason": "简要判断理由",
  "spans": [
    {
      "text": "回复中的幻觉原文片段",
      "hallucination_type": "该片段的幻觉类型",
      "reason": "该片段为何是幻觉"
    }
  ]
}
```

### Prompt 核心要点

1. **角色设定**：你是客服回复质量审核专家，根据知识库检测回复中的幻觉
2. **KB 三种状态明确区分**：
   - "无（系统未接入XX）" → 回复声称已执行 = 幻觉
   - "未标注/未提及" → 回复做肯定性陈述 = 无依据断言
   - 有具体内容 → 逐条比对
3. **输出格式强制**：只输出 JSON，不要任何解释文字，不要 markdown 代码块
4. **关闭思考模式**：API 调用时设置 `reasoning_effort: none` 或等效参数

### JSON 解析多层 Fallback

```
第1层：json.loads() 直接解析
   ↓ 失败
第2层：正则提取第一个 { ... } 再解析
   ↓ 失败
第3层：重试 API 调用（最多 3 次）
   ↓ 失败
标记为"解析失败"，记录原始返回
```

## 五、检出率验证

### 计算指标

- **准确率 (Accuracy)**：整体判断正确的比例
- **精确率 (Precision)**：判为幻觉中真正是幻觉的比例（衡量误报）
- **召回率 (Recall)**：真实幻觉中被检出的比例（衡量漏检）
- **F1 分数**：精确率和召回率的调和平均
- **按类型分组召回率**：每种幻觉类型分别计算

### 与 ground_truth 的映射

检测工具输出 4 类，ground_truth 有 8 类。比对策略：

- **核心指标**：只看 `is_hallucination` 布尔值是否一致
- **类型比对**：通过映射表转换后作为附加分析

| 检测工具类型 | 映射到 ground_truth 类型 |
|-------------|------------------------|
| 事实冲突 | 参数编造 / 信息编造 / 优惠编造 / 政策编造 / 政策偏差 |
| 能力越界 | 能力越界 |
| 无依据断言 | 信息编造 |
| 安全误导 | 安全误导 |

## 六、打包为 .exe 的雷点与对策

### P0 级（不处理则无法运行）

| 雷点 | 后果 | 对策 |
|------|------|------|
| LLM JSON 输出解析失败 | 工具崩溃，无结果产出 | 三层 fallback 解析 + 重试机制 |
| PyInstaller 打包后依赖缺失 | exe 无法启动 | 用 `requests` 替代 `openai` SDK；`.spec` 中显式声明 hiddenimports |
| 输入 JSON 字段校验缺失 | 用户上传格式不对时崩溃 | 启动时校验必需字段（id, user_question, system_reply, knowledge_base），缺失则弹窗提示 |

### P1 级（影响检测质量）

| 雷点 | 后果 | 对策 |
|------|------|------|
| 知识库"无"的 Prompt 歧义 | 能力越界类系统性漏检 | Prompt 中明确区分三种 KB 状态 |
| API 网络错误无处理 | 检测中断，前功尽弃 | 每条 try/except，失败记录错误继续下一条 |
| 中文编码问题 | 输出文件乱码 | 所有文件读写显式指定 `encoding='utf-8'` |

### P2 级（影响用户体验）

| 雷点 | 后果 | 对策 |
|------|------|------|
| API Key 明文泄露 | 安全风险 | 输入框密码掩码，不写入日志/文件，不持久化 |
| Mock 模式预期错位 | 用户误以为检出率代表真实能力 | Mock 结果文件顶部显式标注警告 |
| 文件路径在 exe 中失效 | 找不到资源文件 | 使用 `sys.executable` 获取 exe 目录，文件选择通过 GUI 对话框 |
| 批量调用上下文污染 | 不同条目间交叉影响 | 逐条调用，不批量 |

## 七、GUI 界面设计

### 布局

```
┌─────────────────────────────────────────┐
│  客服回复幻觉检测工具                      │
├─────────────────────────────────────────┤
│  输入文件：  [____________] [选择文件]    │
│  GT文件(可选)：[____________] [选择文件]  │
│                                         │
│  检测模式：  ○ Mock模式  ● 真实API模式   │
│  API Key：   [____________] (密码掩码)   │
│  API URL：   [____________]             │
│  模型名称：  [____________]             │
│                                         │
│  [开始检测]  进度：第 X/20 条            │
│  ████████████░░░░░░░░  60%              │
│                                         │
│  结果预览：                              │
│  ┌─────────────────────────────────┐    │
│  │ h01: 幻觉-事实冲突-高危          │    │
│  │ h02: 幻觉-事实冲突-中危          │    │
│  │ ...                             │    │
│  └─────────────────────────────────┘    │
│                                         │
│  [导出结果]  [导出评估报告]              │
└─────────────────────────────────────────┘
```

### 交互逻辑

1. 用户选择输入 JSON 文件（必需），可选 ground_truth 文件
2. 选择 Mock 模式 → API 配置区隐藏；选择真实 API → 显示 API Key / URL / 模型名输入框
3. 点击"开始检测" → 逐条处理，进度条实时更新
4. 检测完成 → 结果预览区显示摘要，导出按钮可用
5. 导出生成 `detection_result.json`，如有 GT 则同时生成 `evaluation_report.json`
